# Protocols

Small operating procedures for recurring work. These are reference pages, not daily trackers.

## Study session protocol

Use this when you sit down to study and do not want to waste the first 20 minutes deciding what to do.

### When to use

- English practice
- German practice
- CS review
- application writing

### Steps

1. Pick one concrete output for the session.
2. Open only the material needed for that output.
3. Work for 45–60 minutes.
4. Save the output somewhere permanent.
5. Write one short after-action note in Obsidian.

### Output examples

- one corrected paragraph
- one Anki batch
- one paper summary
- one solved exercise set

## English output loop

Use this when the goal is active English improvement, not passive input.

### Steps

1. Choose one input: article, video, paper abstract, or TOEFL prompt.
2. Extract 5–10 useful phrases.
3. Produce one output: summary, short answer, essay paragraph, or recording.
4. Correct only recurring mistakes.
5. Add reusable phrases to Anki or Obsidian.

### Rule

Input is useful only if it creates output.

## Paper review protocol

Use this for quickly understanding a research paper before deciding whether to reproduce it.

### Steps

1. Read title, abstract, and conclusion.
2. Identify task, dataset, metric, and claimed improvement.
3. Read the method section only after the goal is clear.
4. Search for code, dataset availability, and reproduction notes.
5. Write a 10-line summary.

### Decision

Continue only if the paper has enough implementation detail or accessible code/data.

## Paper reproduction protocol

Use this when turning a paper into code.

### Steps

1. Create a minimal baseline first.
2. Reproduce preprocessing before model changes.
3. Match the evaluation metric exactly.
4. Log every mismatch between paper and implementation.
5. Compare against a simple baseline before optimizing.

### Output

- runnable notebook or script
- reproduction notes
- table of expected vs observed results

## Professor email protocol

Use this before contacting a professor or researcher.

### Steps

1. Read at least one recent paper.
2. Write a 3-line paper-specific observation.
3. State your background in one concise paragraph.
4. Ask for one specific opportunity or next step.
5. Keep the email under 180 words unless needed.

### Avoid

- generic praise
- long biography
- asking questions already answered on their website

## Weekly review protocol

Use this once per week, separate from daily Obsidian tracking.

### Questions

1. What moved forward this week?
2. What got stuck?
3. What should be reduced or paused?
4. What is the one priority for next week?
5. Which roadmap needs adjustment?

### Output

A short weekly note in Obsidian and, if needed, one roadmap update.

## Application shortlisting protocol

Use this when evaluating Master's programs.

### Steps

1. Check tuition and language requirements first.
2. Check deadlines and intake season.
3. Check whether the program matches CS/AI/research interests.
4. Check required documents.
5. Mark fit as High, Medium, or Low.

### Output

A shortlist table with status and next action.
